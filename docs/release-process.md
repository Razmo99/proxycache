# Release Process

## Overview

This repository uses `python-semantic-release` for versioning, changelog generation, Git tags, and GitHub Releases.

- `main` produces stable releases.
- `develop` produces prereleases with the `alpha` token, for example `v1.2.0-alpha.1`.
- Every published GitHub Release triggers a GHCR image build.
- Dependabot updates dependencies and GitHub Actions separately; it does not control the application version number.

## Commit Format

Releases depend on Conventional Commits.

- `fix:` produces a patch release.
- `feat:` produces a minor release.
- `feat!:` or a `BREAKING CHANGE:` footer produces a major release.
- `chore(release):` commits are generated by automation and excluded from the changelog.

## Workflows

- [ci.yml](/home/razmo/project/proxycache/.github/workflows/ci.yml): installs the project and runs `pytest`.
- [release.yml](/home/razmo/project/proxycache/.github/workflows/release.yml): runs semantic release on `main` and `develop`.
- [container-release.yml](/home/razmo/project/proxycache/.github/workflows/container-release.yml): publishes multi-arch GHCR images when a GitHub Release is published.

## Container Tags

Stable releases publish:

- the Git tag, for example `v1.4.2`
- semantic version tags, for example `1.4.2`, `1.4`, `1`
- `latest`
- `main`

Prereleases publish:

- the Git tag, for example `v1.5.0-alpha.1`
- the semantic prerelease tag, for example `1.5.0-alpha.1`
- `prerelease`

## Repository Setup

If branch protection prevents `GITHUB_TOKEN` from pushing release commits and tags, create a repository secret named `SEMANTIC_RELEASE_TOKEN` with a token that can push to the release branches.

## Dependabot

Dependabot is configured in [.github/dependabot.yml](/home/razmo/project/proxycache/.github/dependabot.yml).

- Python dependency updates target `develop`.
- GitHub Actions dependency updates target `develop`.
- Docker base image updates target `develop`.
- Project version bumps remain owned by semantic-release, not Dependabot.
